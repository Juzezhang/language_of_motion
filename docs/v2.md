# Language of Motion v2

**v1 is the default and unchanged** — the main README's configs, pretrained checkpoints, and demos are all v1. **v2** changes the audio tokenizer (HuBERT → GLM-4-Voice) and the language-model architecture (T5 encoder-decoder → decoder-only LLM). This page collects the v2 details.

## Architecture — decoder-only LM

Where v1 uses a T5 encoder-decoder (audio on the encoder, motion on the decoder, joined by cross-attention), v2 models everything as **one causal stream**:

```
[BOS] <audio tokens> [SEP] [FACE]<face> [UPPER]<upper> [LOWER]<lower> [HAND]<hand> [EOS]
```

per-part self-delimiting blocks (face is 4× the body token rate). The backbone is a **pretrained** decoder-only LLM (e.g. `Qwen/Qwen3-0.6B`): its original text vocabulary and transformer weights are **preserved**, and the vocabulary is **enlarged** (the same token-embedding-resize idea LoM already uses) with the GLM audio codebook (16384) + the four motion codebooks (face 512, hand/upper/lower 256 each) + structural tokens (BOS / SEP / per-part delimiters / EOS). The cross-entropy loss is taken on the motion side only; face-only samples (see *TFHP* below) additionally mask the body-token positions to `-100`. Starting from a pretrained LLM and training long (≈8–15 epochs) matches or beats the from-scratch encoder-decoder, and **Qwen3 consistently outperforms GPT-2** at matched size. The backbone is a stock HuggingFace `AutoModelForCausalLM`, so no custom flash-attention backend is required.

## Setup

- Download the GLM-4-Voice tokenizer to `model_files/glm-4-voice-tokenizer/` (from `THUDM/glm-4-voice-tokenizer`); the decoder-only backbone (e.g. `Qwen/Qwen3-0.6B`) is pulled from the HF Hub — nothing extra to install beyond `transformers`.
- Tokenize audio with GLM-4-Voice instead of HuBERT, then point the dataset config's `code_path_audio` at `audios_token_glm`:
  ```bash
  python -m scripts.get_speech_code_glm --wav_folder <.../wave16k> --output_dir <.../audios_token_glm>
  ```
- **transformers version:** v2 runs on the **newest** `transformers` — tested on **5.12.1**. The GLM-4-Voice tokenizer needs `transformers>=4.43` and the Qwen3 backbone needs `>=4.51`; since the decoder-only LM is a stock HuggingFace `AutoModelForCausalLM`, it works on any recent release. ⚠️ The paper's **v1 `turbot5` backend requires `transformers<=4.49`**: from 4.50 `PreTrainedModel` stopped auto-inheriting `GenerationMixin` (turbot5 loses `.generate()`), and 5.x removed `transformers.utils.model_parallel_utils` (turbot5 fails to import). To reproduce v1/turbot5, downgrade with `pip install transformers==4.49.0`.

## Root translation

v2 recovers the root translation with ViBES's method — the Global-VAE predicts the root *local* (pelvis-frame) velocity, which is rotated into world space by the pelvis orientation and integrated (`integrate_local_velocity` in `lom/utils/other_tools.py`) — instead of v1's EMAGE-style `velocity2position` (world-velocity integration, height channel taken raw). It uses the ViBES Global-VAE checkpoint at `model_files/pretrained_cpt/VAE_Global_from_Lower54/`.

## TFHP — extra talking-head data for A2M

The audio-to-motion task can be trained on **BEAT2 + TFHP** together. TFHP (`HDTF_TFHP`, from [DiffPoseTalk](https://github.com/DiffPoseTalk/DiffPoseTalk)) is **face-only** (FLAME; no body/SMPL-X), so it is added to A2M with the body (hand/upper/lower) tokens **masked to `-100`** in the loss — it contributes face + audio supervision only, leaving the body head trained on BEAT2 alone. Prepare it as:

- **Per-clip layout** (nested `<speaker>/<session>/<index>`):
  ```
  <TFHP_ROOT>/TOKENS_AGENT_25/face/<spk>/<sess>/<idx>.npy   # face tokens (ViBES face VQ, codebook 512, 25 tok/s)
  <TFHP_ROOT>/audios_token_glm/<spk>/<sess>/<idx>.npy        # GLM-4-Voice audio tokens (12.5 tok/s)
  ```
- **Face tokens:** encode each clip's FLAME face feature (112-D = 6-D head rotation + 6-D jaw + 100 expression — the same layout as v2 BEAT2) with the v2 ViBES face VQ (`scripts/get_face_code_v2.py`).
- **Audio tokens:** `python -m scripts.get_speech_code_glm --wav_folder <.../wav16k> --output_dir <.../audios_token_glm>` (same GLM-4-Voice tokenizer as BEAT2).
- Set the dataset root in `configs/assets.yaml` → `DATASET.TFHP.ROOT`. The A2M config (`configs/config_mixed_stage3_a2m_v2.yaml`) already lists a `TFHP` entry under `datasets:`. TFHP is added to **training only**; evaluation stays on BEAT2. The face-only masking is handled automatically via a `face_only` flag on the task.

## FLAME face visualization (PyTorch3D, adapted from ViBES)

To render the generated co-speech **face** as a FLAME mesh in the ViBES look (HardPhongShader, teal mesh), v2 vendors ViBES's renderer into `lom/render/flame_render.py` — `RenderMesh` + `write_video`, © Xuangeng Chu / [ViBES](https://github.com/). It needs **pytorch3d** and **av**, which are *not* in the LoM model env, so face rendering is split from generation:

```bash
# 1. generate (LoM model env): caches the 112-D face feature _rec_face.npy + _audio.npy
python demo_v2_a2m_face.py --cfg configs/config_mixed_stage3_a2m_v2.yaml --nodebug
# 2. render (any pytorch3d env): FLAME -> mesh -> mp4
PYTHONPATH=. python scripts/render_face_flame.py \
    --rec_face experiments/v2_vis_a2m_face/_rec_face.npy \
    --audio    experiments/v2_vis_a2m_face/_audio.npy \
    --out      experiments/v2_vis_a2m_face/face.mp4 --flame_dir model_files/FLAME2020
```

If pytorch3d is installed in the same env, `demo_v2_a2m_face.py` renders in-process and step 2 is unnecessary. The 112-D face feature is `val_a2m_forward(batch)['rec_face']` = head 6-D + jaw 6-D + 100 expression coeffs; `render_face_sequence(rec_face, flame_dir, out_path, audio=...)` is the reusable entry point.

> v2 changes the audio-token vocabulary, so it requires re-training the language model; the released v1 checkpoints are HuBERT-based and remain the default.
